# 风险解决方案报告

> 针对 constitution v1.0.1 风险评估的 GitHub 调研结果

## 高风险问题（5 项，全部解决）

### 1. Codex CLI Hook stdin 格式 — ✅ 已确认

**调研发现：** 从 `openai/codex` 仓库的 schema 源码（`codex-rs/app-server-protocol/schema/`）确认：

- Codex Hook 事件名是 **camelCase**：`preToolUse`、`permissionRequest`、`postToolUse`、`preCompact`、`postCompact`、`sessionStart`、`userPromptSubmit`、`subagentStart`、`subagentStop`、`stop`
- Claude Code Hook 事件名是 **PascalCase**：`PreToolUse`、`Stop`、`SessionStart` 等
- **语义完全一致**，但大小写不同
- Hook 配置文件格式相同：JSON，`hooks` → event → `[{matcher, hooks: [{type, command}]}]`
- Codex 额外支持：`commandWindows`（Windows 专用命令）、`timeoutSec`、`async`、`statusMessage`
- Codex 的 HookHandlerType 有三种：`command`、`prompt`、`agent`（Claude 只有 `command`）

**解决方案：** Adapter trait 增加 `hook_event_case()` 方法返回 `PascalCase` 或 `camelCase`，Hook 注册时自动转换事件名。Hook 注册逻辑可共享，仅事件名做大小写映射。

### 2. MVP 单工具验证不足 — ✅ 有方案

**调研发现：**
- `kbwo/ccmanager` ⭐1176 — 已支持 8 个工具（Claude Code、Codex CLI、Gemini CLI、Cursor Agent、Copilot CLI、Cline CLI、OpenCode、Kimi CLI）的状态监控
- `SeemSeam/claude_codex_bridge` ⭐3186 — 支持 15 个工具家族的 TUI 工作台
- ccmanager 有"可配置的状态检测策略"和"状态变更 hooks"（idle/busy/waiting_input 三态）

**解决方案：** 阶段一改为 Claude Code + Codex CLI 双工具 MVP。两者 Hook 系统语义一致（仅大小写不同），增量成本仅一个事件名转换。这样能真正验证"多工具统一看板"的核心价值。

### 3. Tailwind v4 稳定性 — ✅ 已解决

**调研发现：** shadcn/ui issue #6668（`feat(components): Add Tailwind v4 theme code generation`）已于 2026-04-05 关闭。shadcn/ui 的主题生成器现在支持选择 Tailwind v4 或 v3，组件兼容性问题已修复。issue #5557（React 19 兼容性）也已关闭。

**解决方案：** Tailwind v4 + shadcn/ui 已完全兼容，无需 escape hatch。宪法约束保持不变。

### 4. Wayland 终端跳转 — 🟡 降级方案

**调研发现：**
- `wlr-foreign-toplevel-management` 协议存在（Wayland 的窗口管理标准），有 C/Zig 实现示例
- 没有成熟的 Rust crate
- Tauri 的 Wayland 支持仍在发展中
- `xdotool` 在 Wayland 下不工作

**解决方案：** Linux Wayland 环境下降级为"仅通知不跳转"。在 adapter 的 `WindowManager` 中检测 Wayland 会话（`$WAYLAND_DISPLAY` 环境变量），Wayland 下跳转操作返回 `Unsupported` 状态，看板 UI 显示"此环境不支持跳转"提示。X11 环境继续使用 xdotool。

### 5. Windows symlink 权限 — ✅ 已解决

**调研发现：** skills-manager-jw 的 `linker.rs` 有完整的 Windows 实现：

```rust
// 检测 symlink 和 Junction（通过 reparse point 标志）
pub fn is_symlink_or_junction(path: &Path) -> bool {
    if let Ok(meta) = path.symlink_metadata() {
        if meta.file_type().is_symlink() { return true; }
        #[cfg(windows)]
        {
            use std::os::windows::fs::MetadataExt;
            const FILE_ATTRIBUTE_REPARSE_POINT: u32 = 0x0400;
            if meta.file_attributes() & FILE_ATTRIBUTE_REPARSE_POINT != 0 { return true; }
        }
    }
    false
}

// Windows 上 Junction 用 remove_dir，symlink 用 remove_file
pub fn remove_symlink_or_junction(path: &Path) -> Result<(), std::io::Error> {
    #[cfg(windows)]
    { return fs::remove_dir(path).or_else(|_| fs::remove_file(path)); }
    #[cfg(unix)]
    { fs::remove_file(path) }
}
```

**解决方案：** Windows 上目录使用 Junction（`mklink /J`），文件使用 copy 模式（带 `.skills-manager-source.json` 元数据）。不依赖 symlink。直接移植 skills-manager-jw 的实现。

## 中风险问题（已解决的项）

### 6. Adapter trait 过早抽象 — ✅ 有方案

**解决方案：** Rust trait 的默认实现（default method）可以解决。只有 `name()` 和 `detect()` 是 required，其余方法返回默认值：
- `hook_config_path()` → `None`（OpenCode 无 Hook）
- `read_hooks()` → `vec![]`
- `subagent_dir()` → `None`
- `mcp_format()` → `McpFormat::Json`（默认）

### 7. Codex Hook 事件名大小写 — ✅ 已解决

见高风险问题 1。在 adapter trait 中增加事件名转换。

### 8. 轮询 vs 文件监听 — ✅ 有方案

**调研发现：** `notify-rs/notify` ⭐3407 — Rust 生态中最成熟的跨平台文件系统通知库，支持 FSEvents（macOS）、inotify（Linux）、ReadDirectoryChangesW（Windows）。

**解决方案：** 双层策略：
- active 会话（Thinking/Processing/Waiting）：50ms 轮询 Hook 事件文件
- idle 会话：5s 低频轮询
- 叠加 `notify` crate 监听会话日志目录变更，文件变化时触发即时刷新

### 9. 安全路径列表不完整 — ✅ 已解决

**调研发现：** amux 源码中有完整的敏感路径列表：

```python
_SENSITIVE_PATHS = {".ssh", ".gnupg", ".aws", ".kube", ".netrc", ".npmrc",
                    ".docker", ".config/gcloud", ".config/gh"}
```

**解决方案：** 直接参考 amux 的列表，写入宪法原则 VII。

### 10. MCP 桥接 PoC — ✅ 不需要

**调研发现：** ccmanager 已经支持 Cursor Agent 的状态监控，且不需要 MCP 插件 — 它通过进程扫描 + 输出解析实现。说明 APP 形态工具的监控不一定需要 MCP 桥接。

**解决方案：** 阶段三先参考 ccmanager 的方式做 Cursor 监控（进程扫描 + 输出解析），MCP 桥接作为增强方案后续评估。

### 11. Kimi Code 接口 — ✅ 已确认

**调研发现：**
- `MoonshotAI/kimi-cli` ⭐9124 — Python CLI 工具，正在演变为 `MoonshotAI/kimi-code` ⭐2966
- 有 VS Code 扩展
- ccmanager 已支持 Kimi CLI 状态监控

### 12. OpenClaw 接口 — ⚠️ 暂未找到

**调研发现：** `openclaw/openclaw` 不是 AI 编程工具。cc-switch 的描述中提到 OpenClaw，但没有找到独立的 OpenClaw AI 编程工具仓库。

**解决方案：** OpenClaw 暂时不在阶段三范围内，待确认其是否存在后再补充。
