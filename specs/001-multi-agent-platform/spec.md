# 功能规格说明：多 Agent 编程工具统一管理平台

**功能分支**：`001-multi-agent-platform`

**创建日期**：2026-07-05

**状态**：草稿

**输入**：用户描述 — 构建一个多 Agent 编程工具统一管理平台，解决两个核心痛点：(1) 多终端窗口中 AI 编程工具的运行状态不清晰，缺乏统一看板和提醒机制；(2) 跨工具重复安装 skill/MCP/插件，版本管理混乱，切换不便。

## 用户场景与测试

### 用户故事 1 — 多工具会话监控看板 (优先级: P1)

用户在多个终端窗口中同时运行 Claude Code 和 Codex CLI 等 AI 编程工具。每个工具的上下文输出都混在各自终端窗口里，用户无法一眼掌握哪个工具在运行、哪个在等待批准、哪个已完成。用户需要一个统一看板，以红绿灯形式聚合展示所有活跃会话的状态，无论会话来自哪个工具。

**优先级理由**：这是整个平台的核心价值 — 解决"多窗口、多工具状态不可见"的痛点。没有这个功能，后续的通知、跳转、管理都无法展开。

**独立测试**：打开 2 个终端分别运行 Claude Code 和 Codex CLI，启动本应用，验证看板中同时显示两个会话且状态正确（运行中/等待/已完成）。

**验收场景**：

1. **给定** 用户有 2 个 Claude Code 会话和 1 个 Codex CLI 会话正在运行，**当** 用户打开看板，**则** 看板显示 3 个会话卡片，每个卡片包含工具名称、项目名称、当前状态（红/黄/绿灯）、最后一条消息预览、运行时长
2. **给定** 某个 Claude Code 会话正在执行工具调用（Processing 状态），**当** 工具调用完成、Claude 发出文本回复，**则** 该会话状态自动从黄色（运行中）变为红色（等待用户输入）
3. **给定** 用户关闭了某个终端窗口中的 Claude Code 会话，**当** 下一次轮询周期到达，**则** 看板中该会话卡片消失或标记为已完成
4. **给定** 用户启动了一个新的 Codex CLI 会话，**当** 下一次轮询周期到达，**则** 看板中自动新增一个会话卡片
5. **给定** 看板中有多个会话，**当** 用户查看，**则** 正在运行的会话排在最前面，等待用户操作的排其次，空闲的排最后
6. **给定** 用户有一个 OpenCode 会话正在运行，**当** 用户打开看板，**则** 看板显示该会话卡片，标记为 OpenCode 类型，状态通过进程扫描 + 数据文件解析获取

---

### 用户故事 2 — 状态变更通知与提醒 (优先级: P2)

用户经常将终端放在后台运行任务，切回其他工作。当 AI 编程工具完成任务、遇到错误或等待用户批准时，用户无法及时知晓，导致任务停滞。用户需要通过声音和桌面通知被告知状态变化，不同状态可配置不同的提醒方式。

**优先级理由**：终端后台运行是核心使用场景，没有通知机制，用户必须频繁切回终端检查状态，严重降低效率。

**独立测试**：启动一个 Claude Code 会话执行任务，切到其他应用，当 Claude 完成任务时验证收到声音和桌面通知。

**验收场景**：

1. **给定** 用户正在其他应用中工作，某个会话从"运行中"变为"等待用户输入"，**当** 状态变化发生，**则** 用户收到桌面通知（含会话名称和状态），并听到提示音
2. **给定** 用户已配置"等待批准"状态使用提示音 A、"任务完成"使用提示音 B，**当** 分别触发这两种状态变化，**则** 用户听到不同的提示音
3. **给定** 用户不希望频繁被打扰，**当** 用户在设置中关闭通知，**则** 不再收到任何声音和桌面通知
4. **给定** 用户暂时不想看到看板窗口，**当** 用户通过全局热键切换，**则** 看板窗口隐藏/显示，但通知功能不受影响

---

### 用户故事 3 — 快速跳转到终端 (优先级: P3)

用户在看板中发现某个会话需要关注（等待批准或已完成），需要快速切换到对应的终端窗口进行操作。手动在多个终端窗口间查找和切换效率低下，尤其在开了 5-10 个窗口时。

**优先级理由**：看板 + 通知解决了"知道哪个需要关注"的问题，跳转解决了"快速到达"的问题，两者配合形成完整闭环。

**独立测试**：在看板中点击一个会话卡片，验证对应的终端窗口被激活并置前。

**验收场景**：

1. **给定** 看板中有一个运行在 iTerm2 中的 Claude Code 会话，**当** 用户点击该会话卡片，**则** iTerm2 被激活，对应的终端标签页被选中并置前
2. **给定** 看板中有一个运行在 tmux 中的 Codex CLI 会话，**当** 用户点击该会话卡片，**则** tmux 中对应的 pane 被选中
3. **给定** 用户的系统环境不支持终端跳转（如 Linux Wayland），**当** 用户点击会话卡片，**则** 显示"此环境不支持跳转"提示，但不影响其他功能

---

### 用户故事 4 — Skill/MCP/插件 统一仓库管理 (优先级: P4)

用户在 Claude Code、Codex CLI、OpenCode、OpenClaw 等多个工具中重复安装相同的 skill、MCP 服务器和插件，占用空间、版本不同步、切换工具时需重新配置。用户希望在一个统一仓库中集中管理这三类资源，然后按需映射到各工具。

**优先级理由**：这是需求二的核心。统一管理 skill（文件映射）、MCP（配置写入）和插件（文件/配置混合），消除跨工具重复安装。

**独立测试**：在统一仓库中安装一个 skill 和一个 MCP 服务器，为 Claude Code 启用 skill、为 Codex CLI 启用 MCP、为 OpenClaw 启用 skill，验证 Claude Code 的 skill 目录包含该 skill、Codex CLI 的 config.toml 包含该 MCP 配置、OpenClaw 的 skills 目录包含该 skill。

**验收场景**：

1. **给定** 统一仓库中有 10 个 skill、3 个 MCP 服务器和 2 个插件，**当** 用户打开资源管理看板，**则** 三类资源分开展示，每条记录显示名称、描述、来源、已分配的工具
2. **给定** 用户为 Claude Code 启用了 3 个 skill，**当** 用户查看 Claude Code 的 skill 目录，**则** 目录中只有那 3 个 skill 的链接（symlink/Junction），原始文件不受影响
3. **给定** 用户为 Codex CLI 启用了一个 MCP 服务器，**当** 启用操作完成，**则** Codex CLI 的 config.toml 中自动写入对应的 [mcp_servers] 配置段（TOML 格式），且 Claude Code 的配置不受影响
4. **给定** 用户在统一仓库中更新了一个 skill 的内容，**当** 更新完成，**则** 所有已启用该 skill 的工具自动获得最新版本（通过 symlink 共享）
5. **给定** 用户为 Claude Code 启用了一个 MCP 服务器，**当** 启用操作完成，**则** Claude Code 的 ~/.claude.json 中自动写入 mcpServers 配置（JSON 格式），且 Codex CLI 的 config.toml 不受影响
6. **给定** 用户为某个工具禁用了一个 MCP 服务器，**当** 禁用操作完成，**则** 该工具的配置文件中移除对应条目，但统一仓库中的 MCP 配置记录保留
7. **给定** 用户为 OpenCode 启用了一个 MCP 服务器，**当** 启用操作完成，**则** OpenCode 的 opencode.json 中写入 mcp 配置（JSONC 格式，含 type/command/environment 字段）
8. **给定** 用户为 OpenClaw 启用了 2 个 skill，**当** 启用操作完成，**则** OpenClaw 的 ~/.openclaw/skills/ 目录中出现那 2 个 skill 的链接，且 OpenClaw 的 openclaw.json 中 mcp.servers 配置不受影响

### 用户故事 5 — 预设组一键切换 (优先级: P5)

用户在不同项目中需要不同的资源组合。例如"前端开发"可能需要 3 个 skill + 1 个 MCP 服务器 + 1 个插件，"后端开发"需要另外的组合。逐个启用/禁用效率低，用户希望能将 skill、MCP 和插件打包为命名预设组，一键为某个工具切换整组配置。

**优先级理由**：在统一仓库基础上提升管理效率。预设组不限于 skill，而是 skill + MCP + 插件的混合组合，直接作为一个整体应用到目标工具。

**独立测试**：创建一个包含 2 个 skill + 1 个 MCP 的预设组，为 Codex CLI 一键应用，验证 Codex CLI 同时获得这 2 个 skill 的链接和 1 个 MCP 的配置。

**验收场景**：

1. **给定** 用户创建了"前端开发"预设组（含 2 个 skill + 1 个 MCP 服务器 + 1 个插件），**当** 用户在 Codex CLI 的配置中点击该预设，**则** 3 项资源同时为 Codex CLI 启用 — skill 目录出现 2 个链接，config.toml 出现 1 个 MCP 配置段
2. **给定** "前端开发"预设已对 Codex CLI 激活，**当** 用户再次点击该预设，**则** 3 项资源同时从 Codex CLI 禁用（链接移除、配置条目移除），但统一仓库中的原始资源不受影响
3. **给定** 同一个预设组，**当** 用户分别应用到 Codex CLI 和 Claude Code，**则** 两个工具各自获得对应格式的配置（Codex 用 TOML，Claude 用 JSON），且互不影响
4. **给定** 用户从系统托盘菜单操作预设，**当** 用户点击托盘菜单中的预设名称，**则** 预设生效，无需打开完整应用窗口

### 用户故事 6 — 子 Agent 级资源分配 (优先级: P6)

使用 Hermes、OpenCode 等多 Agent 工具时，工具内部有多个子 Agent（如 researcher、coder、reviewer），每个子 Agent 有不同职责。子 Agent 的资源分配应该是工具级分配的子集 — 即子 Agent 只能使用其所属工具已启用的资源范围内的子集。

**优先级理由**：这是工具级分配的精细化扩展。核心逻辑是"工具级分配圈定范围 → 子 Agent 级分配在范围内筛选"。不单独存在，依赖于用户故事 4/5 先完成工具级分配。

**独立测试**：为 Hermes 启用 20 个 skill，然后为其中的 researcher 子 Agent 分配 5 个、coder 子 Agent 分配 10 个，验证子 Agent 只能使用分配的子集，且不超过工具级范围。

**验收场景**：

1. **给定** 用户已为 Hermes 启用了 20 个 skill，**当** 用户为 Hermes 的 researcher 子 Agent 分配其中 5 个，**则** researcher 的 skill 目录中只有那 5 个 skill 的链接，其余 15 个不出现
2. **给定** 用户尝试为 researcher 子 Agent 分配一个未在工具级启用的 skill，**当** 执行分配操作，**则** 系统拒绝并提示"该 skill 未在 Hermes 的工具级分配中启用，无法分配给子 Agent"
3. **给定** 用户创建了一个"研究专用"预设组（含 3 个 skill），**当** 用户将该预设应用到 researcher 子 Agent，**则** 只有 researcher 的配置发生变化，coder 和其他子 Agent 不受影响，且预设中的 skill 必须在工具级范围内
4. **给定** 用户在工具级禁用了一个 skill，**当** 该 skill 被禁用，**则** 所有子 Agent 中该 skill 的分配也自动移除

## 功能需求

### FR-1: 会话发现与状态检测

1. 系统必须自动发现所有正在运行的 AI 编程工具会话（MVP 支持 Claude Code、Codex CLI、OpenCode 和 OpenClaw）。同一工具的 CLI 形态与桌面 APP 形态（如 Codex CLI 与 Codex 桌面 APP）通过进程名区分为不同会话，均纳入监控；APP 形态不支持终端跳转，仅展示状态与通知
2. 系统必须实时检测每个会话的运行状态，状态分为：运行中（生成/工具执行）、等待用户输入、空闲、已完成
3. 有 Hook 系统的工具必须通过 Hook 事件捕获状态变更；无 Hook 的工具回退到进程扫描
4. 状态检测失败时不得崩溃，必须降级为"未知"状态并继续运行

### FR-2: 统一监控看板

5. 看板必须聚合展示所有活跃会话，每个会话以卡片形式显示
6. 每个会话卡片必须显示：工具名称、项目名称、状态指示灯（红/黄/绿）、最后消息预览、运行时长、CPU 使用率
7. 看板必须按状态优先级排序：需要关注的（等待）排最前，运行中的排其次，空闲的排最后
8. 系统托盘必须显示聚合状态指示灯：任一会话等待 → 红灯，有运行中 → 黄灯，全部空闲 → 绿灯

### FR-3: 通知与提醒

9. 会话状态变更时必须触发桌面通知，包含会话名称和新状态
10. 用户必须能为不同状态配置不同的提示音
11. 用户必须能全局开启/关闭通知
12. 通知必须可点击，点击后跳转到对应会话

### FR-4: 快速跳转

13. 点击会话卡片必须激活对应的终端窗口或标签页
14. macOS 上必须支持 iTerm2、Terminal.app、kitty、WezTerm 和 tmux
15. 不支持跳转的环境必须显示明确提示，不影响其他功能

### FR-5: 扩展资源统一仓库（三层映射架构）

16. 系统必须维护一个全局仓库（SSOT），集中管理三类资源：Skill（文件/目录）、MCP 服务器（配置条目）、插件（文件/配置混合）
17. **三层映射架构**：
    - **Layer 1（SSOT）**：`~/.mam/skills/` 存放所有 skill 原始文件；`~/.mam/mcps/` 存放 MCP 配置；`~/.mam/plugins/` 存放插件原始文件
    - **Layer 2（工具级激活目录）**：`~/.mam/active/<tool>/` 存放该工具已启用的 skill 链接；MCP 和插件直接写入工具配置文件
    - **Layer 3（子 Agent 级激活目录）**：`~/.mam/active/<tool>/<subagent>/` 存放该子 Agent 已启用的 skill 链接；仅 Hermes 和 OpenCode 等支持子 Agent 独立 skill 目录的工具有此层
18. Skill 必须通过符号链接（Unix）或 Junction/copy（Windows）从 Layer 1 映射到 Layer 2/3
19. MCP 服务器必须按目标工具的配置格式写入（Claude Code=JSON mcpServers、Codex CLI=TOML mcp_servers、OpenCode=JSONC mcp）
20. 插件必须根据类型处理：文件型插件用 symlink 映射，配置型插件写入工具配置
21. MCP 服务器和插件可为每个工具独立启用/禁用（每个 资源×工具 组合有独立状态）；Skill 不支持单独为工具启用/禁用，只能通过预设组整体应用到目标工具（见 FR-6），从而避免手动单独分配与预设组移除之间的冲突
22. 在统一仓库中更新资源后，所有已启用的工具必须自动获得最新版本
23. 禁用资源时必须从工具的配置/目录中移除映射，但统一仓库中的原始记录保留

### FR-6: 预设组管理（含 Skill+MCP+插件组合）

24. 用户必须能创建命名预设组，每个预设组可包含任意组合的 skill + MCP + 插件
25. 用户必须能一键为某个工具或子 Agent 激活/停用整组预设 — 同时处理 skill 链接、MCP 配置、插件映射
26. 同一预设组应用到不同工具时，必须自动适配各工具的配置格式
27. 预设组操作必须可通过系统托盘菜单执行，无需打开完整窗口
28. 预设组应用过程中，若某项资源启用失败，保留已成功的项并将失败项报告给用户手动处理（不自动回滚已成功的项）

### FR-7: 子 Agent 级分配（工具级范围的子集）

29. 对于支持子 Agent 独立 skill 目录的工具（Hermes、OpenCode），系统必须能为每个子 Agent 独立分配资源，但分配范围不得超出工具级已启用的资源
30. 子 Agent 级配置必须与工具级配置独立，工具级变更时子 Agent 级自动同步（如工具级禁用则子 Agent 级也移除）
31. 子 Agent 级也支持预设组操作（含 skill 组合），但预设组中的资源必须在工具级范围内；skill 的子 Agent 级分配同样只能通过预设组完成
32. 对于不支持子 Agent 独立 skill 目录的工具（Claude Code、Codex CLI），子 Agent 级分配功能降级为"仅 UI 记录"，不实际映射到文件系统

### FR-8: 安全与透明

33. 所有 skill/MCP/插件的安装路径和权限必须对用户可见
34. 敏感路径（SSH 密钥、环境变量、凭据等）不得被读取或暴露
35. 所有监控必须在本地完成，不得向外传输任何数据
36. 安全扫描仅检查 skill 文件路径是否落在允许的目录范围内（防止路径穿越攻击），不检查文件内容

## 成功标准

1. 用户能在单一看板中同时监控 Claude Code、Codex CLI、OpenCode 和 OpenClaw 的会话状态，无需切换终端窗口
2. 会话状态变更后 3 秒内，用户收到桌面通知
3. 用户点击看板中的会话后 1 秒内，对应终端窗口被激活
4. 在统一仓库中安装一个 skill 后，10 秒内可完成到 2 个以上工具的映射
5. 用户创建预设组后，一键切换可在 5 秒内完成整组 skill 的启用/禁用
6. 应用启动时间不超过 3 秒（冷启动，20 个活跃会话场景下）
7. 状态轮询不影响系统其他操作的流畅度（CPU 占用 < 5%）
8. 所有功能完全离线运行，不依赖任何外部网络服务

## 关键实体

| 实体 | 说明 | 关键属性 |
|------|------|---------|
| 会话 (Session) | 一次 AI 编程工具的运行实例 | ID、工具类型、项目路径、状态、PID、CPU |
| Agent 工具 (AgentTool) | 被监控的 AI 编程工具 | 名称、进程标识、Hook 支持、MCP 格式、Skill 目录、子 Agent 列表、是否支持子 Agent 独立目录 |
| 扩展资源 (Extension) | Skill/MCP/插件三类资源的统一抽象 | 类型(Skill/MCP/Plugin)、名称、来源、版本、已分配工具、支持工具列表（Claude/Codex/OpenCode/OpenClaw） |
| 预设组 (Preset) | 扩展资源的命名组合（可含 Skill+MCP+插件） | 名称、资源列表（含类型+引用+兼容性标记）、目标工具/子Agent |
| 子 Agent (SubAgent) | 多 Agent 工具内部的子角色 | ID、名称、所属工具、资源分配范围（工具级子集）、独立 skill 目录路径 |

## 假设

1. 用户已在机器上安装了至少一种支持的 AI 编程工具（Claude Code、Codex CLI、OpenCode 或 OpenClaw）
2. macOS 为主要开发和测试平台，Linux 和 Windows 为后续扩展
3. 用户有基本的终端使用能力
4. 支持的工具会持续更新其 Hook/日志/MCP 格式，平台需要跟进适配
5. 阶段一（MVP）聚焦 Claude Code + Codex CLI + OpenCode + OpenClaw 四工具，包含完整资源管理（skill/MCP/插件统一仓库 + 预设组 + 子 Agent 分配），后续阶段逐步扩展更多工具
6. 技术栈约束已在项目宪法（`.specify/memory/constitution.md` v1.2.0）中定义
