## 二、需求一技术细节

### 2.1 claude-control — 状态识别与通知的标杆

**架构：** Electron + Next.js 桌面应用，macOS 专属

**核心技术机制：**

#### Hook 事件系统
- 在 `~/.claude/settings.json` 中注册 hook 脚本 `~/.claude-control/hooks/status-hook.sh`
- 监听 7 种事件：SessionStart / SessionEnd / Stop / UserPromptSubmit / PermissionRequest / SubagentStart / PostToolUseFailure
- hook 脚本从 stdin 读取 JSON，用 grep/sed 提取字段（无 jq 依赖），写入 `~/.claude-control/events/<pid>.json`
- 事件文件按 PPID（Claude 进程 PID）命名，实现进程级会话追踪

```
关键路径: src/lib/hooks-installer.ts → src/lib/hooks-reader.ts → src/lib/discovery.ts
```

#### 状态分类器（五态模型）
```
src/lib/status-classifier.ts

输入: { pid, jsonlMtime, cpuPercent, hasError, isAskingForInput, hasPendingToolUse }
输出: working | idle | waiting | errored | finished

规则:
- pid === null → finished（进程已退出）
- hasError → errored
- CPU > 15% 或 (JSONL 最近写入 + CPU > 5%) → working
- hasPendingToolUse + JSONL 超过 APPROVAL_SETTLE_MS 未更新 → waiting（等待批准）
- isAskingForInput → waiting（Claude 在问问题）
- 进程活着但无活动 → idle
```

#### 进程发现与会话关联
```
src/lib/discovery.ts + src/lib/terminal/detect.ts

1. ps -eo pid,ppid,%cpu,comm 构建进程树
2. 查找 comm === "claude" 的 PID
3. 通过 PPID 关联终端进程
4. hook 事件的 transcript_path → JSONL 会话文件
5. JSONL fallback: 找项目目录下最新 .jsonl 文件
```

#### 终端跳转（多终端支持）
```
src/lib/terminal/detect.ts

支持终端: iTerm2 / Terminal.app / Ghostty / kitty / WezTerm / Warp / Alacritty / cmux
方式: AppleScript (macOS) 控制终端窗口聚焦
tmux: send-keys 远程操作，无需聚焦终端
```

#### 通知系统
```
src/hooks/useNotificationSound.ts  — Web Audio API 生成双音 chime (880Hz A5 + 1174.66Hz D6)
src/hooks/useDesktopNotification.ts — macOS 原生通知 + Canvas 生成状态颜色图标
                                      waiting=蓝 / idle=琥珀 / finished=锌灰
                                      通知可点击跳转到对应会话
```

### 2.2 ccboard — Rust 双前端架构的标杆

**架构：** Rust workspace，4 个 crate

```
ccboard (binary) — CLI 入口，模式路由
├── ccboard-tui (lib) — Ratatui TUI，12+ 个 tab
├── ccboard-web (lib) — Axum API + Leptos 前端
└── ccboard-core (lib) — 共享数据层、业务逻辑
    ├── ccboard-types — 类型定义
    └── ccboard-ffi   — FFI 绑定
```

**核心设计：**

#### 单二进制双前端
```rust
// Arc<DataStore> 在 TUI 和 Web 间零序列化共享
let store = Arc::new(DataStore::new(config)?);
match cli.command {
    TUI => run_tui(Arc::clone(&store)),
    Web => run_web(Arc::clone(&store)),
    Both => { tokio::spawn(run_web(Arc::clone(&store))); run_tui(store) }
}
```

#### DataStore 中央状态
```rust
pub struct DataStore {
    // 低竞争: RwLock
    stats: RwLock<Option<StatsCache>>,
    settings: RwLock<MergedConfig>,
    mcp_config: RwLock<Option<McpConfig>>,
    live_hook_sessions: RwLock<LiveSessionFile>,

    // 高竞争: DashMap (per-key 锁)
    sessions: DashMap<String, Arc<SessionMetadata>>,

    // LRU 缓存
    session_content_cache: Cache<String, Vec<String>>,

    // 事件总线
    event_bus: EventBus,

    // SQLite 元数据缓存 (持久化)
    metadata_cache: Option<Arc<MetadataCache>>,
}
```

#### 懒加载策略
- 启动时只提取 JSONL 首行+尾行的 metadata（3350 个会话 <300ms）
- 详情视图按需加载完整内容
- SQLite 缓存带来 89x 加速

#### 多工具会话识别
- 识别第三方会话标记: Cursor `[Cu]` / Codex `[Cx]` / OpenCode `[Oc]`
- 通过解析各工具的 JSONL 会话文件格式

### 2.3 amux — tmux 集成与多工具编排的标杆

**架构：** Python 单文件服务器 + Go 二进制 + Web/PWA/iOS

**核心技术：**

#### tmux 会话管理
- 所有 Agent 会话运行在 tmux 中
- `~/.amux/sessions/` 存储会话状态
- 通过 `tmux send-keys` 实现远程操作（批准/拒绝/发送消息）
- 自愈机制：自动压缩上下文 → 重启 → 重放最后消息

#### 多工具集成
```python
# ~/.amux/ 目录结构
sessions/       — 会话状态
logs/           — 日志
memory/         — 持久记忆
board/          — Kanban 看板
channels/       — Agent 间消息线程 (JSONL)
notes/          — 跨会话笔记
transcripts/    — 会话备份
notifications.json — 通知配置
```

#### Agent 编排
- Kanban 看板: SQLite CAS 原子任务认领
- Channels: 会话间 1:1 实时聊天 + @mentions
- REST API: 发送消息 / peek 输出 / 认领任务

### 3.1 HarnessKit — 扩展管理覆盖面最广

**架构：** Tauri 2 + React 前端 + Rust 核心

```
crates/
├── hk-core/     — 核心库
│   ├── adapter/ — 9 个 agent 适配器
│   ├── auditor/ — 安全审计引擎 (18 条规则)
│   ├── deployer.rs — 跨 agent 部署
│   ├── kits/    — Kit 打包安装系统
│   ├── manager.rs — 扩展管理 (toggle/install/uninstall)
│   ├── scanner.rs — 扫描发现
│   └── store.rs   — SQLite 存储
├── hk-desktop/  — Tauri 桌面壳
├── hk-web/      — Web 前端
└── hk-cli/      — CLI 入口
```

**核心机制：AgentAdapter Trait**

```rust
pub trait AgentAdapter: Send + Sync {
    fn name(&self) -> &str;
    fn base_dir(&self) -> PathBuf;
    fn detect(&self) -> bool;           // 检测工具是否安装
    fn skill_dirs(&self) -> Vec<PathBuf>;
    fn mcp_config_path(&self) -> PathBuf;
    fn hook_config_path(&self) -> PathBuf;
    fn plugin_dirs(&self) -> Vec<PathBuf>;
    fn read_mcp_servers(&self) -> Vec<McpServerEntry>;
    fn read_hooks(&self) -> Vec<HookEntry>;
    fn read_plugins(&self) -> Vec<PluginEntry>;
    fn hook_format(&self) -> HookFormat;
    fn mcp_format(&self) -> McpFormat;
    fn translate_hook_event(&self, event: &str) -> Option<String>;
    // ... 还有 rules/memory/settings/subagent/workflow 发现方法
}
```

**格式转换系统：**
```
McpFormat:
  McpServers  — JSON {"mcpServers": {...}} (Claude/Gemini/Cursor)
  Servers     — JSON {"servers": {...}} (Copilot/VS Code)
  Toml        — TOML [mcp_servers.<name>] (Codex)
  Opencode    — JSON {"mcp": {...}} (OpenCode)
  HermesYaml  — YAML mcp_servers: (Hermes)

HookFormat:
  ClaudeLike  — {"hooks": {"Event": [{"matcher": "...", "hooks": ["cmd"]}]}}
  Cursor      — {"version": 1, "hooks": {...}}
  Copilot     — {"version": 1, "hooks": {"event": [{"type": "command", "bash": "cmd"}]}}
  Windsurf    — {"hooks": {"event": [{"command": "cmd"}]}}
  HermesYaml  — YAML config.yaml
```

**部署器 (deployer.rs)：**
- `deploy_skill`: 递归拷贝目录（跳过 .git 和 symlink）
- `deploy_mcp`: 格式化写入各 agent 的 MCP 配置
- `resolve_command_path`: 将裸命令解析为绝对路径（GUI agent 不继承 shell PATH）
- `sanitize_mcp_name`: MCP 名称清理（Codex 要求 `^[a-zA-Z0-9_-]+$`）

**安全审计 (auditor/)：**
- 18 条静态分析规则
- Trust Score 0-100: Safe(80+) / Low Risk(60-79) / Needs Review(<60)
- 五维度权限扫描: 文件系统 / 网络 / Shell 命令 / 数据库 / 环境变量
- 每个 agent 的副本独立审计

### 3.2 skills-manager (jiweiyeah) — symlink 同步机制

**架构：** Tauri 2 + React 19 + Rust 后端

```
src-tauri/src/
├── services/
│   ├── linker.rs       — symlink/copy 双模式链接
│   ├── detector.rs     — 工具检测 (rayon 并行)
│   ├── scanner.rs      — skill 扫描
│   ├── config_manager.rs — 配置管理
│   └── file_ops.rs     — 文件操作
├── commands/
│   ├── sync.rs          — 同步状态检查
│   ├── skills.rs       — skill CRUD
│   └── tools.rs        — 工具管理
└── models/
    ├── skill.rs        — Skill 数据模型
    ├── tool.rs         — 工具定义 (29+)
    └── config.rs       — 配置结构
```

**LinkerService 核心机制：**

```
双模式:
1. Symlink 模式 — 创建符号链接 (Unix) 或 Junction (Windows)
2. Copy 模式   — 复制文件 + 写入 .skills-manager-source.json 元数据

LinkStatus:
  Valid       — 链接正常
  Broken      — 链接断裂
  WrongTarget — 指向错误源
  NotALink    — 不是链接
  Missing     — 不存在

同步检查:
  对每个 (skill, tool) 组合检查 link 状态
  应启用但链接异常 → 报告 issue
  应禁用但链接存在 → 报告 issue
```

**工具检测 (detector.rs)：**
- 29+ 内置工具定义 (SUPPORTED_TOOLS)
- rayon 并行检测所有工具
- 优先使用用户保存的自定义路径
- fallback 到默认路径 + 备用路径
- 检测: 目录存在 + CLI 可用性

### 3.3 skills-manager (xingkongliang) — Presets 预设组

**架构：** Rust/Tauri 桌面应用

**核心特性：**
- 统一 skill 仓库: `~/.skills-manager`
- **Presets 系统**: 将 skill 分组为命名预设，一键激活/停用整组
- 系统托盘集成: 从托盘菜单操作 preset
- 三级 Workspace: Global / Agent / Project
- 技能更新检查 + 托盘通知

```
Presets 工作流:
1. 用户创建 preset (如 "前端开发组")
2. 将相关 skill 加入 preset
3. 在某个 agent 的 workspace 中点击 preset pill
4. 一键为该 agent 启用/停用整组 skill

托盘状态:
  Empty    — preset 无 skill
  Inactive — preset 有 skill 但未激活
  Partial  — 部分激活
  Active   — 全部激活
```

### 6.5 补充结论

1. **agent-sessions 是最佳起步参考** — 技术栈完全匹配，架构干净（AgentDetector trait + 纯消息状态判断 + sysinfo 进程发现 + AppleScript 终端跳转），代码量小，容易理解和扩展

2. **tauri-app-template 可直接作为项目脚手架** — 已预配置 shadcn/ui + Tailwind v4 + 全局快捷键 + 自动更新 + 国际化 + 暗色模式

3. **HarnessKit 的 AgentAdapter trait 是需求二的最佳参考** — 5 种扩展类型 + 5 种 MCP 格式 + 5 种 Hook 格式 + 安全审计，覆盖面最广

4. **建议的项目启动路径**：tauri-app-template 脚手架 → 移植 agent-sessions 的监控逻辑 → 移植 HarnessKit 的 adapter 模式 → 补充 Presets 篮子分配
